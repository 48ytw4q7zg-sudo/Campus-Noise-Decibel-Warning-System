## 审核报告

### 维度 1:功能完整性

- **issue-1** [严重度: 高]:P0 缺少噪声原始数据的基本查询和查看能力。当前 P1-5"噪声数据列表与筛选"是唯一的数据查询入口，但被放在 P1。核心业务流"数据采集→存储→查看→阈值判断→告警"在数据采集后缺少查看链路——用户采集了噪声数据后无法在 P0 阶段查看原始记录详情，只能通过 P0-4 仪表盘看汇总。这会影响 P0 端到端跑通后的验证(无法确认数据是否真的正确入库)。
  - **位置**:§三 P0 清单 + §四 P1-5
  - **修复建议**:将 P1-5"噪声数据列表与筛选"升级为 P0-7，包含基本的分页列表查询 + 按时间/功能区简单筛选。P1 可保留更高级的多条件组合筛选(关键词等)。

- **issue-2** [严重度: 中]:P0-2"噪声数据采集与存储"描述为"接收/模拟噪声传感器数据"，但未明确该功能的 API 形态——是传感器主动 POST 推送?还是系统定时拉取?如果是模拟数据，是前端随机生成还是后端定时生成?模糊的 API 边界会让 Phase 3 /api-designer 和 Phase 4 后端开发时接口设计不清晰。
  - **位置**:§三 P0-2 说明列
  - **修复建议**:在 P0-2 说明中明确"提供 POST /api/noise/record 接口接收传感器上报数据，同时提供后端模拟数据生成能力(Scheduled 定时任务生成模拟噪声数据用于演示)"。

### 维度 2:功能合理性(教学可行性 + 跟约束冲突)

- **issue-3** [严重度: 中]:Python Flask ccswitch 配置服务作为额外技术栈(python + flask + anthropic SDK)超出了教师项目模板的 SpringBoot 3 + Vue 3 + MySQL 标准栈范围。虽然学生主动要求增加以支持阈值热更新，但需在标定卡和技术栈表中明确标注为"可选加分项 · 非教师模板强制"，避免 Phase 4-5 的 feature-coder/service-coder 等命令在读取 CLAUDE.md 技术栈表时误认为 Flask 是必修依赖而导致生成错误代码。
  - **位置**:§一 技术栈行
  - **修复建议**:将技术栈行改为"SpringBoot 3.5.14 + Vue 3.5.34 + MySQL 8.4 + MyBatis-Plus 3.5.15(基础必做) + Python Flask ccswitch 配置服务(P2 可选加分项 · 用于阈值热更新)"。

- **issue-4** [严重度: 低]:§二量化锚点"数据字段数=4"过于简化。研究报告中的 4 字段(时间点、地点、分贝值、人工标注)是研究数据集字段，实际生产系统需要更多字段:噪声类型(交谈/施工/体育等)、持续时间(秒)、数据来源设备ID、是否异常等。§二量化锚点与实际数据库设计会有偏差，建议区分"研究数据集字段"和"系统数据库字段"。
  - **位置**:§二 量化锚点表
  - **修复建议**:保留"研究数据字段=4"说明，同时补充"系统数据库字段预计 8-10 个(含 create_time/update_time/设备ID/噪声类型等)"。

### 维度 3:角色与权限清晰度

- **issue-5** [严重度: 高]:P0 功能清单中的所有功能均未标注各功能对应的角色权限。例如 P0-6"功能区配置管理"显然应该是管理员专属，P0-4"实时噪声监测仪表盘"应该是所有登录用户可查看。缺少角色权限标注会导致 Phase 3 api-designer 无法确定接口权限设计(哪些接口需要 @PreAuthorize)，也会影响 Phase 4 后端生成 Controller 时权限注解缺失。
  - **位置**:§三 P0 清单全表
  - **修复建议**:在 P0 功能清单表中新增一列"角色权限"，标注每项功能的访问角色。示例:P0-1 无需登录(公开)· P0-2 管理员可写+普通用户可读 · P0-6 管理员专属。

- **issue-6** [严重度: 中]:"普通用户"角色在校园噪声场景中的具体权限范围未定义。当前 §一 写"普通用户 / 管理员"，但"普通用户"可能包含不同职级:是只能查看仪表盘?还是也能确认/处理告警?还是能配置部分阈值?从业务合理性出发，"普通用户"应定义为"可查看所有监测数据+告警记录+统计图表，但不可修改系统配置(功能区/阈值规则)"，管理员可做全部操作。
  - **位置**:§一 JWT 角色行
  - **修复建议**:在 §一 JWT 角色行补充角色权限简述:"普通用户(可查看噪声数据+告警+统计图表)/管理员(可管理功能区配置+阈值规则+系统设置)"。

### 维度 4:业务闭环性

- **issue-7** [严重度: 低]:user 核心实体在所有优先级(P0/P1/P2)中均缺少"个人信息更新/修改密码"功能。虽然这不是噪声业务的核心，但作为全栈项目的基本用户管理能力，至少应有个入口(P1 或 P0-1 扩展)。当前只有注册和登录，用户无法修改密码——这在真实项目中是基本的账号管理需求。
  - **位置**:§三 P0-1 说明列
  - **修复建议**:在 P0-1 说明中补充"含个人信息查看+修改密码"，或在 §四 新增 P1-6"个人信息管理(修改密码+基本信息编辑)"。

### 维度 5:P0/P1/P2 划分合理性 ⭐

- **issue-8** [严重度: 中]:P0 接口数统计偏紧。现有 6 个 P0 功能标注的接口数合计:3+3+2+2+2+2=14，在"12-15"范围内。但如果执行 issue-1 的修复(把 P1-5 数据查询升级为 P0)，会增加 1-2 个接口使总数达到 15-16。实际上 16 个 P0 接口在教学范围内(P0 8-12 接口是典型值但非硬上限，且这个系统有噪声数据采集+阈值判断的特殊算法逻辑)，建议将 §二量化锚点 P0 接口数更新为 14-16。
  - **位置**:§二 量化锚点表 "P0 接口数"行
  - **修复建议**:将 P0 接口数从"12-15"调整为"14-16"(反映实际 6-7 个 P0 功能的接口数)。同时建议在应用修复后将各功能的接口数和表数重新核算，确保标定卡数据准确。

## 修改决策记录(答辩素材 · 学生答辩时可直接念这段)

> 📌 **本段由 §二 应用修复后回填** · 学生答辩时念清楚"原标定/调整后/理由"即可 · 评委会认可这是企业开发的需求二次评审环节(加分点)

### 调整 1:P1-5 "噪声数据列表与筛选"升级为 P0-7
- **原标定**:P1-5"噪声数据列表与筛选"放在 P1 应做功能，P0 阶段只有 P0-4 仪表盘可看汇总，无原始记录查询入口
- **调整后**:新增 P0-7"噪声数据列表与筛选"（分页列表+按时间/功能区/分贝范围/异常状态筛选），原 P1-5 改为"噪声数据高级筛选与导出"（在 P0-7 基础上升级复杂组合筛选+CSV导出）
- **理由**:R-00 issue-1——核心业务流"数据采集→存储→查看→阈值判断→告警"在采集后缺少查看链路，用户无法验证数据是否正确入库，必须闭环

### 调整 2:P0/P1/P2 全部功能清单补充"角色权限"列
- **原标定**:P0/P1/P2 功能清单表头仅有"编号/功能/说明/表数/接口数/页面数/工时"7 列，未标注角色权限
- **调整后**:全部三级功能清单新增"角色权限"列，明确每项功能的访问角色（公开/普通用户可查看/管理员专属/后台自动等）
- **理由**:R-00 issue-5——缺少角色权限标注会导致 Phase 3 api-designer 无法确定接口 @PreAuthorize 设计，Phase 4 后端生成 Controller 时权限注解缺失

### 调整 3:明确 P0-2 数据采集 API 形态
- **原标定**:P0-2 说明为"接收/模拟噪声传感器数据"，未区分 API 形态
- **调整后**:明确双通道——POST /api/noise/record 接收传感器上报 + @Scheduled 定时任务每5分钟自动生成模拟噪声数据(用于演示)
- **理由**:R-00 issue-2——API 形态模糊会导致 Phase 3 接口设计和 Phase 4 后端实现时困惑

### 调整 4:技术栈标注 Flask 为"P2 可选加分项"
- **原标定**:技术栈行写"SpringBoot 3 + Vue 3 + MySQL + MyBatis-Plus + Python Flask(ccswitch)"
- **调整后**:改为"SpringBoot 3.5.14 + Vue 3.5.34 + MySQL 8.4 + MyBatis-Plus 3.5.15（基础必做）+ Python Flask ccswitch 配置服务（P2 可选加分项 · 用于阈值热更新）"
- **理由**:R-00 issue-3——Flask 不在教师模板标准栈中，需明确标注为加分项以免 Phase 4-5 的 feature-coder 等命令误读为必修依赖

### 调整 5:明确"普通用户"角色权限范围
- **原标定**:§一 JWT 角色行仅写"普通用户 / 管理员"，权限范围模糊
- **调整后**:补充为"普通用户（可查看噪声数据+告警+统计图表）/ 管理员（可管理功能区配置+阈值规则+系统设置+用户管理）"
- **理由**:R-00 issue-6——角色权限范围不明会导致后续 Phase 4 Service 层权限校验设计不准确

### 调整 6:P0-1 补充个人信息查看+修改密码
- **原标定**:P0-1 仅含"用户注册、登录、token签发与校验"，接口数3个
- **调整后**:补充"个人信息查看+修改密码"，接口数调整为4，工时调整为2.5h
- **理由**:R-00 issue-7——全栈项目基本用户管理能力必备，user 实体缺 CRUD 的 Update 能力

### 调整 7:§二量化锚点数据字段数区分研究/系统视角
- **原标定**:数据字段数仅写"4（时间点、地点、分贝值、人工标注）"
- **调整后**:改为"研究数据集4字段 → 系统数据库预计8-10字段（含create_time/update_time/设备ID/噪声类型/持续时间等）"
- **理由**:R-00 issue-4——区分研究数据和系统数据库字段，避免 Phase 2 db-designer 认为只需4个字段

### 调整 8:P0 接口数更新为 14-16
- **原标定**:P0 接口数"12-15"
- **调整后**:更新为"14-16（因P1-5噪声数据查询已升级为P0-7，增加1-2个查询接口）"
- **理由**:R-00 issue-8——P0-7 新增后总接口数增加，需更新量化锚点与实际一致

## 修复行动建议

**按严重度排序的修复优先级**:

1. **[高] issue-1 + issue-5**:最优先修复——P1-5 升级为 P0-7 噪声数据查询(确保核心业务流闭环)+ P0 功能清单补充角色权限列(确保下游 Phase 3-4 权限设计准确)。这两个 issue 直接影响到 Phase 3 API 设计和 Phase 4 代码生成的质量。

2. **[中] issue-2 + issue-3 + issue-6**:次要修复——明确数据采集 API 形态(避免 Phase 3 接口设计模糊)、标注 Flask 为可选加分项(避免下游命令误读)、明确普通用户权限范围(避免角色权限设计出错)。

3. **[低] issue-4 + issue-7 + issue-8**:可择机修复——补充用户个人信息管理、细化数据字段、更新接口数统计。这些属于细节完善，不影响 P0 核心流程跑通。

> 建议高+中严重度 issue 都通过 `/scoping-reviewer 应用修复` 一次性修复，低严重度 issue 在应用修复时顺手修正。
