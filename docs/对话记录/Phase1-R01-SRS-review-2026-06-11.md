## 审核报告

### 维度 1：完整性

- **issue-1** [严重度：高]：P0-1 业务规则 §② 声明"管理员可禁用用户（P2 扩展）"，但 P2 功能清单（P2-1~P2-5）中没有用户管理/禁用用户的对应功能
  - **位置**：PRD §3 P0-1 业务规则 · §3 P2-1~P2-5
  - **推演**：P0 阶段留了扩展承诺 → Phase 4 coder 会预留 disable 字段但不暴露接口 → P2 阶段学生找不到对应 feature → 要么漏做要么临时塞进某页面 → 破坏切片顺序
  - **修复建议**：二选一 ——（a）从 P0-1 业务规则中移除"管理员可禁用用户（P2 扩展）"，改为"管理员禁用用户功能不在本期范围内"；或（b）在 P2 新增 P2-6"用户管理（管理员禁用/启用用户）"，补全功能说明

### 维度 2：一致性

- **issue-2** [严重度：高]：P0-3 业务规则 §① 时段标签映射漏掉"早读（7:30-8:00）"，与 P1-4 时段枚举 {早读, 上课, 午休, 活动, 晚自修, 夜间静校, 用餐时段, 非用餐时段} 不一致，也与研究报告 §3.3.1 阈值规则字典（图书馆早读 43dB）不一致
  - **位置**：PRD §3 P0-3 业务规则 · P1-4 业务规则
  - **推演**：P0-3 时间映射为 `8:00-12:00=上课、12:00-14:00=午休、14:00-18:00=上课、18:00-22:00=活动/晚自修、22:00-7:30=夜间静校` → 7:30-8:00 时段无标签 → P0-3 阈值判断时该时段噪声记录落入未定义状态 → 兜底到 default_threshold（48.3dB 的均值）而非研究报告设计的 43dB → 图书馆早读时段漏报风险
  - **修复建议**：P0-3 业务规则 §① 补充早读时段映射：`7:30-8:00=早读`，使时段标签集合与 P1-4 枚举一致

- **issue-3** [严重度：中]：P0-3 主流程 §5 和 P0-5 主流程 §1 都声称"Service 层自动向 alert_log 表写入告警记录"，对同一写入操作的职责归属描述重复且模糊——到底是 P0-3 的 Service 写、还是 P0-5 的 Service 写、还是两者是同一段代码？
  - **位置**：PRD §3 P0-3 主流程 §5 · P0-5 主流程 §1
  - **推演**：Phase 4 两个 coder 分别实现 P0-3 和 P0-5 → 各自写了一条 alert_log insert → 同一条异常产生两条告警记录 → 告警列表翻倍
  - **修复建议**：明确告警写入的唯一入口——建议在 P0-3 主流程 §5 改为"标记异常后，触发 P0-5 告警创建流程"，P0-5 主流程 §1 保留"P0-3 标记异常后，由本功能的 AlertService 统一写入 alert_log"

### 维度 3：可行性

- 无 issue。P0 共 7 个功能，对照标定卡 §二量化锚点 P0 接口 14-16 个/页面 6-7 个/工时约 18h，大学生 3 周可完成。单功能主流程步骤均 ≤8 步，无外部 API 依赖。

### 维度 4：明确性

- **issue-4** [严重度：中]：P0-2 手动录入路径中 device_id 格式未定义——传感器上报时 device_id 由硬件决定，但管理员手动录入时应填什么？如果不填，与"设备 ID 非空"的数据约束矛盾
  - **位置**：PRD §3 P0-2 业务规则
  - **修复建议**：在 P0-2 业务规则 §③ 补充"管理员手动录入时 device_id 自动填充 'MANUAL_' + 时间戳前缀，区分传感器来源"

- **issue-5** [严重度：低]：P0-2 分贝范围设为 0-120 dB(A)——0 dB(A) 为听觉阈值（消声室才可能），实际校园场景最低约 30 dB(A)；上限 120 dB(A) 超出研究报告数据范围（35.2-85.7 dB(A)）
  - **位置**：PRD §3 P0-2 业务规则
  - **修复建议**：将校验范围收紧为 20-120 dB(A)，或标注"0-120 为声级计物理量程，实际校园场景 30-95 dB(A)"

- **issue-6** [严重度：中]：P0-3 异常流程 §③ 写"乐观锁或数据库去重（通过 noise_record.id + is_abnormal 已标记检查）"——表述模糊，"已标记检查"是 IS NOT NULL 还是 =1？P0-2 状态机定义 is_abnormal 初始为 null → 0 或 null → 1
  - **位置**：PRD §3 P0-3 异常流程
  - **修复建议**：改为"检查 noise_record.is_abnormal IS NOT NULL，已判断则跳过（幂等保证）"

#### 7.1 删除依赖推演：逐实体列推演链

| 被删实体 | 引用方（外键） | 推演场景 | PRD 是否覆盖 | 结论 |

| noise_record | alert_log.noise_record_id | 假设提供删除接口，删除已有告警关联的噪声记录 → alert_log 外键报错 | P0-5 声明"独立保留"，但 noise_record 无删除入口 → 实际不会触发 | ✅ 无风险 |

#### 7.2 NULL / 空集合推演：逐可空字段列推演链

| 实体.字段 | NULL 语义 | 推演场景 | PRD 是否定义 | 判定 |
|----------|---------|---------|-------------|------|
| noise_record.is_abnormal | 未判断 | 查询列表时 is_abnormal=NULL 的记录 | P0-7 定义"显示为'待判断'" | ✅ |
| alert_log.confirmed_by | 未确认 | 按确认人筛选时如何处理 NULL | 未定义 | 见下方 issue |
| area_config.description | 可空 | 列表展示时描述为空 → 显示空列或占位符 | 未定义 | ⚠️ 低风险 |
| area_config.default_threshold | 理论上非空 | 如果 DBA 误将其设为 NULL → P0-3 兜底链断裂 | 未定义 | 见 issue-13 |

- **issue-13** [严重度：中]：P0-3 阈值兜底链为 `threshold_rule 缺失 → area_config.default_threshold`，但 default_threshold 若为 NULL（DBA 误操作 / 初始未配置），无进一步兜底——P0-3 阈值判断将抛 NPE 或误标所有记录
  - **位置**：PRD §3 P0-3 业务规则 · P0-6 业务规则
  - **推演**：DBA 直接操作数据库，将"图书馆"的 default_threshold 误设为 NULL → 图书馆某时段无 threshold_rule → P0-3 取 default_threshold 得 NULL → Java 拆箱 NPE → 全局异常处理器返回 500 → 该时段所有噪声记录判断中断
  - **修复建议**：在 P0-3 业务规则 §③ 补充："若 threshold_rule 和 default_threshold 均不可用，使用全局默认值 55 dB(A)（GB 3096-2008 1 类区昼间标准），并记录 WARN 日志"

#### 7.3 并发 / 重复操作推演：逐状态变更操作列推演链

| 操作 | 推演场景 | PRD 处理 | 判定 |
|------|---------|---------|------|
| P0-1 注册 | 同一用户名并发注册 | 数据库唯一索引防重 | ✅ |
| P0-3 阈值判断 | 同一条 noise_record 并发判断 | 乐观锁/IS NOT NULL 检查 | ✅ |
| P0-5 告警确认 | 两个管理员同时确认同一条 | 乐观锁，先到先得 | ✅ |
| P0-6 功能区修改 | 两个管理员同时改同一功能区 | 乐观锁 version 字段 | ✅ |
| P1-4 规则修改 | 两个管理员同时改同一规则 | 乐观锁 | ✅ |
| P2-2 AI 分类 | 两条定时任务线程同时分类同一条 noise_record | "以最后一次覆盖" —— 无并发控制 | ⚠️ 见 issue-14 |

- **issue-14** [严重度：中]：P2-2 并发/重复操作处理为"以最后一次分类结果覆盖"，未采用乐观锁或幂等机制——虽然 P2 可选且 AI 分类非关键路径，但"覆盖"模式在教学中示范了错误的并发处理
  - **位置**：PRD §3 P2-2 异常流程
  - **推演**：定时任务线程 A 和线程 B 同时扫描到同一条未分类记录 → A 分类为"交谈"写入 → B 分类为"体育活动"覆盖 → A 的结果被静默覆盖 → 无任何日志或告警
  - **修复建议**：改为"通过 noise_record.id + noise_type IS NULL 条件更新（CAS），先到先得，后续跳过（幂等）"
