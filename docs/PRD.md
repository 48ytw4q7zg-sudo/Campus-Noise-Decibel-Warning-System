# 校园噪音分贝预警员系统 - 需求规格说明书

## 1. 项目概述

校园噪音分贝预警员系统是一个基于"业务规则+统计自适应+混合阈值"动态模型的校园噪声监测预警平台。系统面向普通用户和管理员两类角色，覆盖图书馆、食堂、操场、宿舍四大核心功能区，通过传感器数据采集 → 动态阈值计算 → 异常检测 → 告警推送的完整业务闭环，实现校园噪声的精准识别与实时预警。目标异常检测准确率 ≥92%，误报率 ≤5%，为校园管理者提供科学、可量化的噪声管控工具。

## 2. 用户角色定义

- **普通用户** · 可查看噪声监测仪表盘、噪声历史数据、告警记录列表、统计图表；不可修改系统配置和用户管理
- **管理员** · 拥有全部权限：管理功能区配置、阈值规则配置、系统设置、用户管理；可确认/处置告警；可导入导出数据

无权限访问时后端返回 403 + 前端 ElMessage 提示"无权限"；已登录用户访问无权限页面时前端路由守卫拦截并跳回首页。

## 3. 功能需求列表

### P0-1 · 用户注册登录 + JWT

- **实现优先级**：P0 必做
- **描述**：用户注册、登录、JWT token 签发与校验、个人信息查看、修改密码
- **前置条件**：无（注册/登录无需登录；个人信息查看需登录）
- **主流程**：
  1. 新用户访问注册页，填写用户名、密码、角色（普通用户/管理员），提交注册
  2. 系统校验用户名唯一性，BCrypt 加密密码后存入数据库，返回注册成功
  3. 已注册用户访问登录页，填写用户名+密码，提交登录
  4. 系统校验密码，签发 JWT token（含 userId + role），返回 token
  5. 登录后用户可查看个人信息（用户名、角色、注册时间），可修改密码（需输入原密码）
- **异常流程**：
  - ① 输入校验类：用户名为空或已存在 → 返回 1001"用户名已存在"；密码长度 < 6 位 → 返回校验错误
  - ② 数据关联约束类：用户有噪声记录/告警记录等依赖数据时，暂不支持注销（教学简化：仅提供修改密码，不提供账号注销功能）
  - ③ 并发/重复操作类：同一用户名并发注册 → 数据库唯一索引防重，后提交者返回"用户名已存在"
  - ④ 权限边界类：普通用户只能查看和修改自己的个人信息，不能查看/修改其他用户
- **业务规则**：
  - ① 数据约束：用户名 2-20 字符、密码 6-32 字符、角色取值 {"普通用户", "管理员"}
  - ② 状态机：用户状态 {正常, 禁用}，本 P0 仅支持"正常"状态（管理员禁用用户功能不在本期范围内，后续如需扩展需同步补充用户管理模块）

### P0-2 · 噪声数据采集与存储

- **实现优先级**：P0 必做
- **描述**：提供 API 接收传感器上报噪声数据，同时后端定时任务每 5 分钟自动生成模拟噪声数据用于演示；数据存入数据库
- **前置条件**：系统运行中（定时任务自动触发）；传感器上报需提供设备标识
- **主流程**：
  1. 传感器通过 POST /api/noise/record 上报数据（时间点、功能区、分贝值、设备 ID）
  2. 系统校验数据合法性（分贝范围 0-120 dB(A)），存入 noise_record 表
  3. 后端 @Scheduled 定时任务每 5 分钟执行一次：随机选择功能区、生成 35-85 dB(A) 范围内的模拟分贝值，写入 noise_record 表
  4. 新数据入库后自动触发 P0-3 阈值判断流程
- **异常流程**：
  - ① 输入校验类：分贝值超出 0-120 范围 → 返回 400 参数错误；功能区不在四大功能区之列 → 400
  - ② 数据关联约束类：无（数据采集无关联删除场景）
  - ③ 并发/重复操作类：同一设备同一秒内重复上报 → 业务上允许（多次采样可能有相同值），不做幂等去重
  - ④ 权限边界类：传感器上报无需认证（内部接口），前端手动录入需管理员权限
- **业务规则**：
  - ① 数据约束：分贝范围收紧为 20-120 dB(A)（20 为实际校园场景下限，120 为声级计物理量程上限），功能区 ∈ {图书馆, 食堂, 操场, 宿舍}，设备 ID 非空；管理员手动录入时 device_id 自动填充 "MANUAL_" + 时间戳前缀（如 MANUAL_20260611_143000），区分传感器来源

### P0-3 · 基于业务规则的动态阈值判断

- **实现优先级**：P0 必做
- **描述**：按"功能区+时段"二维规则字典计算动态阈值，超阈值标记异常；Service 层自动运行，新噪声数据入库后触发
- **前置条件**：noise_record 表有新数据写入，threshold_rule 表有已配置的规则
- **主流程**：
  1. 新噪声数据写入后，Service 层提取该记录的功能区和时间戳
  2. 根据时间戳确定时段标签（早读/上课/午休/活动/晚自修/夜间静校等）
  3. 以"功能区+时段"为键，查询 threshold_rule 表获取对应阈值
  4. 将当前分贝值与阈值比较：超过阈值则标记 noise_record.is_abnormal = 1
  5. 异常记录写入 alert_log 表（触发 P0-5 告警流程）
- **异常流程**：
  - ① 输入校验类：无（系统内部调用，无用户输入）
  - ② 数据关联约束类：threshold_rule 表中对应功能区+时段的规则被删除 → 使用该功能区的默认阈值（从 area_config 表读取 default_threshold）作为兜底
  - ③ 并发/重复操作类：同一条 noise_record 被并发判断多次 → 通过 is_abnormal IS NOT NULL 检查，已判断则跳过（幂等保证）

- **业务规则**：
  - ① 数据约束：阈值范围 0-120 dB(A)；时段标签由后端根据时间戳自动计算：7:30-8:00=早读、8:00-12:00=上课、12:00-14:00=午休、14:00-18:00=上课、18:00-22:00=活动/晚自修、22:00-7:30=夜间静校
<!-- R-01-issue-2: 已修复 - 时段映射已补充"7:30-8:00=早读"，与P1-4时段枚举{早读,...}一致，也与研究报告§3.3.1图书馆早读43dB对齐 -->
  - ② 状态机：噪声记录异常标记 {null=未判断, 0=正常, 1=异常}，状态只允许 null→0 或 null→1，不允许回退
  - ③ NULL 语义：threshold_rule 表中某功能区某时段无规则 → 使用 area_config.default_threshold 兜底；若 default_threshold 也为 NULL（DBA 误操作或初始未配置），使用全局默认值 55 dB(A)（GB 3096-2008 1 类区昼间标准），并记录 WARN 日志

  - ④ 教学简化声明：业务规则字典的阈值参考研究报告§3.3.1 预定义值，非动态学习调整；时段划分按标准校园作息，不支持自定义作息表

### P0-4 · 实时噪声监测仪表盘

- **实现优先级**：P0 必做
- **描述**：各功能区实时分贝值展示、当前阈值显示、异常状态指示灯；数据每 10 秒自动刷新
- **前置条件**：用户已登录，noise_record 表有数据
- **主流程**：
  1. 用户访问首页仪表盘（/）
  2. 系统查询各功能区最新一条噪声记录的分贝值和异常状态
  3. 系统查询各功能区当前时段对应的阈值
  4. 前端以 2×2 卡片布局展示四个功能区：分贝值（大字体）+ 当前阈值 + 状态指示灯（绿=正常/红=异常）
  5. 每 10 秒自动轮询刷新数据
  6. 点击任意功能区卡片可跳转至该功能区的详细噪声数据页
- **异常流程**：
  - ① 输入校验类：无
  - ② 数据关联约束类：某功能区无噪声数据 → 卡片显示"暂无数据"，指示灯灰色
  - ③ 并发/重复操作类：无（只读查询，无并发风险）
  - ④ 权限边界类：所有登录用户均可查看
- **业务规则**：
  - ① 数据约束：分贝值显示保留 1 位小数；阈值取当前时段的业务规则阈值
  - ② 状态机：指示灯状态 {绿=正常, 红=异常, 灰=无数据}
  - ③ NULL 语义：无
  - ④ 教学简化声明：每 10 秒轮询（非 WebSocket 实时推送），数据刷新间隔内可能有短暂延迟
- **API 形态**：
  - `GET /api/dashboard/overview` → `Result<List<DashboardAreaVO>>`（各功能区实时状态概览）
  - `GET /api/dashboard/area/{location}` → `Result<AreaDetailVO>`（某功能区详细信息）
- **关联页面**：DashboardPage

- **实现优先级**：P0 必做

### P0-6 · 功能区配置管理

- **实现优先级**：P0 必做
- **描述**：管理员管理四大功能区的基础信息（名称、噪声敏感度、默认阈值范围）；支持查看、修改
- **前置条件**：管理员已登录
- **主流程**：
  1. 管理员访问功能区配置页（/area-config）
  2. 系统展示四大功能区列表（图书馆、食堂、操场、宿舍），每行显示：名称、噪声敏感度（高/中/低）、默认阈值上限（dB）、描述
  3. 管理员可编辑各功能区的默认阈值上限和敏感度等级
  4. 修改后保存，系统更新 area_config 表
- **异常流程**：
  - ① 输入校验类：默认阈值超出 0-120 范围 → 400；敏感度取值不在 {高, 中, 低} → 400
  - ② 数据关联约束类：功能区被 threshold_rule 引用时仍允许修改默认阈值（默认阈值仅作兜底，不影响已有规则）；删除功能区 → 拒绝（该功能区已有噪声数据和告警记录，提示"该功能区有 N 条关联数据，不可删除"）
  - ③ 并发/重复操作类：两个管理员同时修改同一功能区 → 乐观锁（version 字段），后提交者提示"数据已被修改，请刷新后重试"
  - ④ 权限边界类：普通用户不可访问此功能
- **业务规则**：
  - ① 数据约束：功能区名称固定为 {图书馆, 食堂, 操场, 宿舍} 四大类（不可新增/删除类型，仅修改参数）；噪声敏感度 {高=1, 中=2, 低=3}
  - ② 状态机：功能区状态 {启用, 停用}，停用后该功能区的噪声数据仍接收但不参与阈值判断
  - ③ NULL 语义：描述字段可空
  - ④ 教学简化声明：四大功能区为固定集合（对齐研究报告），不支持动态新增/删除功能区类型；默认阈值参考研究报告 §3.3.1 预定义值
- **API 形态**：
  - `GET /api/area/list` → `Result<List<AreaConfigVO>>`（功能区列表）
  - `PUT /api/area/{id}` → `Result<Void>`（修改功能区配置，管理员专属）
- **关联页面**：AreaConfigPage

- **主流程**：

### P1-1 · 统计自适应阈值计算

- **实现优先级**：P1 应做
- **描述**：基于滑动窗口技术，实时计算窗口内噪声分贝值的均值 μ 和标准差 σ，动态阈值上下限 = μ ± k × σ；支持配置窗口大小和 k 值
- **前置条件**：noise_record 表有足够数据（窗口大小以上），area_config 表有配置的窗口参数
- **主流程**：
  1. 后台定时任务（或噪声数据入库后触发）对每个功能区独立计算
  2. 取当前时间点前窗口大小内的所有正常噪声数据
  3. 计算滑动窗口内的均值 μ 和标准差 σ
  4. 生成动态阈值上限 = μ + k × σ，下限 = μ - k × σ
  5. 将当前分贝值与动态阈值上下限比较，超出范围标记为异常
  6. 滑动窗口随新数据进入而实时滑动
- **异常流程**：
  - ① 输入校验类：窗口大小配置 < 5 或 > 100 → 业务异常 2001"窗口大小超出合理范围"；k 值 < 1 或 > 5 → 2002"k 值超出合理范围"
  - ② 数据关联约束类：窗口内数据不足 → 回退到业务规则阈值（P0-3 的规则字典）
  - ③ 并发/重复操作类：两次计算同时进行 → 计算结果以最后一次为准（覆盖式更新，不做版本控制）
  - ④ 权限边界类：管理员可配置窗口大小和 k 值，普通用户不可修改
- **业务规则**：
  - ① 数据约束：窗口大小默认 15 分钟（图书馆/宿舍）或 10 分钟（食堂/操场）；k 值默认 2（图书馆/宿舍）或 3（食堂/操场）；阈值上限不超过 120 dB(A)，下限不低于 20 dB(A)
  - ② 状态机：无独立状态机
  - ③ NULL 语义：无
  - ④ 教学简化声明：滑动窗口为固定大小（不按功能区时段动态调整窗口）；σ=0 时（窗口内全相同值）使用硬编码 ±3dB 幅度
- **API 形态**：
  - `GET /api/threshold/adaptive/current?location=图书馆` → `Result<AdaptiveThresholdVO>`（查询某功能区当前统计自适应阈值）
  - `PUT /api/threshold/adaptive/config` → `Result<Void>`（配置窗口大小和 k 值，管理员专属）
- **关联页面**：ThresholdConfigPage（0.5 页 · 自适应阈值参数配置区域）

  1. 管理员在系统设置页配置报告计划：报告周期（日/周/月）、生成时间、报告接收邮箱
  2. 定时任务按配置的计划自动执行：统计周期内各功能区噪声数据（平均分贝、异常率、告警次数、趋势对比）
  3. 生成报告（HTML 或 Markdown 格式），存储到 report 表
  4. 用户可在报告页查看已生成的报告列表，点击下载
  5. 教学简化：不实现真实邮件发送，仅在系统内展示报告
- **异常流程**：
  - ① 输入校验类：报告周期不合法 → 400；邮箱格式错误 → 400（教学简化：不实际发送，仅存储）
  - ② 数据关联约束类：报告引用的噪声数据被删除 → 报告已生成的内容不变
  - ③ 并发/重复操作类：同一周期报告被定时任务重复触发 → 检查该周期是否已有报告，有则跳过
  - ④ 权限边界类：管理员配置报告计划；所有用户可查看已生成报告
- **业务规则**：
  - ① 数据约束：报告周期 ∈ {日, 周, 月}；日报告生成时间默认每天 8:00，周报告默认每周一 8:00，月报告默认每月 1 日 8:00
  - ② 状态机：报告状态 {生成中, 已生成, 生成失败}
  - ③ NULL 语义：报告接收邮箱可空（教学简化：不实际发送邮件）
  - ④ 教学简化声明：报告生成为系统内展示，不实现邮件发送；报告格式为结构化数据+简单文本描述，不含图表嵌入
- **API 形态**：
  - `GET /api/report/list?pageNum=1&pageSize=10` → `Result<IPage<ReportVO>>`（报告列表分页）
  - `GET /api/report/{id}` → `Result<ReportDetailVO>`（报告详情）
  - `POST /api/report/generate` → `Result<Void>`（手动触发报告生成，管理员专属）
  - `PUT /api/report/config` → `Result<Void>`（配置报告计划，管理员专属）
- **关联页面**：StatisticsPage（报告入口）+ SystemSettingsPage（报告配置）

### P2-4 · 多维度统计分析

- **实现优先级**：P2 可选
- **描述**：按功能区/时段/噪声类型多维度交叉分析，含 ECharts 热力图
- **前置条件**：用户已登录，noise_record 和 alert_log 表有足够数据，P1-3 基础可视化已实现
- **主流程**：
  1. 用户在统计分析页使用多维度分析面板：选择分析维度（X 轴：时段/功能区/Y 轴：平均分贝/异常率/告警次数）、选择图表类型（柱状图/热力图/雷达图）
  2. 系统查询聚合数据，返回前端渲染 ECharts 图表
  3. 热力图展示：X 轴=时段（6 个时段标签），Y 轴=功能区（4 个），颜色深浅=平均分贝
  4. 雷达图展示四大功能区在 5 个维度的评分（平均分贝、异常率、噪声波动、告警频率、噪声类型多样性）
- **异常流程**：
  - ① 输入校验类：选择的维度组合无数据 → 图表显示"暂无数据"占位
  - ② 数据关联约束类：无（只读查询）
  - ③ 并发/重复操作类：无
  - ④ 权限边界类：所有登录用户均可查看
- **业务规则**：
  - ① 数据约束：聚合时间粒度默认 1 小时（可按需调整）；热力图颜色梯度 5 级
  - ② 状态机：无
  - ③ NULL 语义：噪声类型为空的数据不参与噪声类型维度的统计
  - ④ 教学简化声明：分析维度为固定预设组合（非自由拖拽 olap 多维度分析）；数据聚合为后端 SQL GROUP BY 实现（非实时 olap 引擎）
- **API 形态**：
  - `GET /api/statistics/multi-dim?xDim=&yDim=&dateFrom=&dateTo=` → `Result<MultiDimVO>`（多维度聚合数据）
  - `GET /api/statistics/heatmap?dateFrom=&dateTo=` → `Result<HeatmapVO>`（热力图数据）
  - `GET /api/statistics/radar?dateFrom=&dateTo=` → `Result<RadarVO>`（雷达图数据）
- **关联页面**：StatisticsPage（多维度分析区域）

### P2-5 · ccswitch 配置热更新面板

- **实现优先级**：P2 可选
- **描述**：管理面板查看和修改 ccswitch 实时配置（模型名、API 端点、阈值参数等），支持一键重载
- **前置条件**：管理员已登录，ccswitch 配置服务运行中
- **主流程**：
  1. 管理员访问系统设置页（/settings）的 ccswitch 面板
  2. 系统调用 ccswitch 服务 /api/health 接口，展示当前配置状态：AI 模型名、base URL、配置来源、运行时间
  3. 管理员可一键点击"重载配置"按钮，系统调用 ccswitch 服务 /api/config/reload 接口
  4. 重载结果反馈到前端（新模型名、是否切换成功）
  5. 展示所有当前环境变量键列表
- **异常流程**：
  - ① 输入校验类：无（无用户输入参数）
  - ② 数据关联约束类：ccswitch 服务不可用 → 面板显示"ccswitch 服务未连接"，重载按钮禁用
  - ③ 并发/重复操作类：多次点击重载 → 按钮在请求进行中时禁用（前端防抖）
  - ④ 权限边界类：管理员专属
- **业务规则**：
  - ① 数据约束：无
  - ② 状态机：配置状态 {正常, 重载中, 重载失败}
  - ③ NULL 语义：无
  - ④ 教学简化声明：ccswitch 面板仅展示和重载配置，不支持在此面板直接修改 settings.json（需手动编辑）；ccswitch 服务独立于主系统运行（Port 5000），需单独启动
- **API 形态**：
  - `GET /api/ccswitch/status` → `Result<CcswitchStatusVO>`（查询 ccswitch 配置状态）
  - `POST /api/ccswitch/reload` → `Result<CcswitchReloadVO>`（触发配置重载，管理员专属）
- **关联页面**：SystemSettingsPage（ccswitch 面板区域）

### 4.1 统一错误码规范

所有接口返回的 `Result<T>` 中 `code` 字段遵循以下分段规范，各模块业务异常按码段分配：

| 码段 | 模块 | 示例 |
|:--:|------|------|
| 1xxx | 用户模块 | 1001 用户名已存在、1002 密码错误 |
| 2xxx | 阈值/统计模块 | 2001 窗口大小超出合理范围、2002 k 值超出合理范围 |
| 3xxx | 规则配置模块 | 3001 该功能区此时段已有规则 |
| 4xxx | 告警模块 | 4001 告警已被处理（并发确认冲突） |
| 5xxx | 数据导入导出模块 | 5001 文件格式不支持、5002 表头不匹配 |
| 6xxx | 功能区配置模块 | 6001 功能区不存在、6002 功能区有依赖数据不可删除 |
| 7xxx | 系统/ccswitch 模块 | 7001 ccswitch 服务不可用、7002 配置重载失败 |

HTTP 状态码通用约定：200=成功（code=200）、400=参数校验错误（code=对应业务码）、401=未登录、403=无权限、404=资源不存在、500=服务器内部错误。
